# JDK 19 面试要点（2022）

## 1. 虚拟线程 Virtual Threads（预览，JDK21 转正）

- **轻量级线程**，由 JVM 调度，阻塞 I/O 时自动 unmount，不占用 OS 线程
- `Thread.startVirtualThread()`、`Thread.ofVirtual().start()`
- **不是**更快 CPU 计算；适合**高并发 I/O**
- 面试：**与平台线程、线程池**对比；**不要**在 synchronized 块里做长时间阻塞（早期 pin 问题，需了解演进）

## 2. 结构化并发 Structured Concurrency（预览）

- 子任务生命周期绑定到父作用域，失败时一起取消
- JDK19 中属于 incubator/预览方向，后续 JDK API 多次演进；面试重点是思想，不是死背某一版方法名

### StructuredTaskScope 核心思想

- **作用域化**：一组并发子任务被放进一个明确的 scope，像 `try-with-resources` 一样有开始和结束边界
- **父子关系**：父任务 fork 出子任务；父作用域结束前，子任务必须完成、失败或被取消
- **避免任务泄漏**：父方法返回/抛异常后，不应该留下还在后台跑的“孤儿任务”
- **统一异常处理**：子任务失败可以被父作用域集中感知，而不是散落在多个 `Future.get()` 里
- **统一取消传播**：一个子任务失败时，可以取消其他还没完成的兄弟任务
- **配合虚拟线程**：虚拟线程让“一个子任务一个线程”成本很低；结构化并发让这些线程更容易管理

### 和 ExecutorService + Future 的区别

| 对比 | `ExecutorService + Future` | `StructuredTaskScope` |
|------|-----------------------------|------------------------|
| 生命周期 | 子任务可能比父方法活得更久 | 子任务绑定到 scope |
| 失败处理 | 需要逐个 `Future.get()` 才发现 | scope 可以集中处理 |
| 取消 | 手动 `cancel(true)`，容易漏 | scope 策略可统一取消 |
| 可读性 | 并发关系分散 | 并发关系在一个代码块里 |
| 目标 | 通用任务提交工具 | 管理“一组相关子任务” |

### 常见策略

- **全部成功才成功**：多个子任务都成功后，父任务组合结果；任一失败则整体失败
- **任一成功即可**：比如同时请求多个镜像服务，谁先成功用谁，取消其他任务
- **失败即取消其他**：请求 A/B/C 三个服务，A 失败后 B/C 没必要继续跑
- **等待全部完成**：需要收集所有结果或所有错误时使用

### 典型场景

```java
// 伪代码，表达思想：同一个业务请求里并发查多个依赖。
try (var scope = StructuredTaskScope.open()) {
    var user = scope.fork(() -> loadUser());
    var order = scope.fork(() -> loadOrder());
    var coupon = scope.fork(() -> loadCoupon());

    scope.join();

    return new Page(user.get(), order.get(), coupon.get());
}
```

- `user/order/coupon` 都属于同一个 scope
- `scope.join()` 表示父任务等待子任务完成
- 离开 scope 时，不应该残留未管理的子任务

### 面试坑

- **结构化并发不是为了让 CPU 计算更快**；它主要提升并发任务的生命周期管理和异常/取消语义
- **不是替代所有线程池**；适合“一次请求里的一组相关子任务”，不适合长期后台任务
- **和虚拟线程是互补关系**：虚拟线程解决线程成本，结构化并发解决线程组织方式
- **JDK19 API 不是最终形态**；JDK19 是早期阶段，JDK21/JDK25 仍有 API 演进

## 常见考题

**1. 虚拟线程为什么能创建百万级？载体线程（carrier）是什么？

**答：**
- **虚拟线程**：JVM 调度的轻量线程，栈可挂载/卸载，阻塞 I/O 时**释放底层载体线程**，不占用 OS 线程等待。
- **载体线程（carrier）**：实际执行字节码的 **平台线程**（内核线程），数量通常 ≈ CPU 核数；大量虚拟线程复用少量 carrier。
- 百万级指**内存占用低**（约 KB 级栈）+ **调度成本低**，非「百万核并行」。

**2. 为什么虚拟线程不适合 CPU 密集任务？

**答：**
- CPU 密集时虚拟线程**一直占用 carrier** 执行计算，无法 unmount，等价于占满平台线程池，**无法通过切换提升吞吐**。
- 大量计算仍受 CPU 核数限制；应用**平台线程池 + 合理并行度**（如 `ForkJoinPool`）更合适。
- 长时间 `synchronized` 可能 **pin** 在 carrier 上（JDK 持续优化），I/O 阻塞场景才是主战场。

**3. 结构化并发解决什么问题？

**答：**
- 把**一组相关子任务**视为一个工作单元：父作用域结束时，子任务全部完成或**全部取消**。
- 解决「子线程泄漏、异常吞掉、取消传播困难」——比手动 `ExecutorService` + `Future.cancel` 更清晰。
- 典型：并发调多个 API，一个失败则其余自动取消（JDK21+ 预览，25 转正演进）。

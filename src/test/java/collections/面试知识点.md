# Java 集合（Collection）面试要点

> 覆盖 **List、Set、Map、Queue/Deque** 高频考点；与 `jdk9` 不可变集合工厂、`threading` 中 CHM 章节可交叉阅读。

## 1. 体系结构

```
Iterable
  └── Collection
        ├── List（有序、可重复）
        ├── Set（不重复）
        └── Queue / Deque
  Map（键值对，独立接口）
```

- **Collection** 与 **Map** 是两条线；`Map` 不继承 `Collection`
- **fail-fast**：迭代时结构被修改 → `ConcurrentModificationException`（除并发容器迭代器）

## 2. List

| 实现 | 底层 | 查 | 头插 | 尾插 | 面试点 |
|------|------|----|------|------|--------|
| `ArrayList` | 动态数组 | O(1) | O(n) | 均摊 O(1) | 默认容量 10，扩容约 1.5 倍 |
| `LinkedList` | 双向链表 | O(n) | O(1) | O(1) | 也实现 `Deque` |
| `Vector` | 线程安全数组 | — | — | — | 已过时，用 `Collections.synchronizedList` 或并发 List |

- **`subList`** 是原列表视图，结构性修改原列表会导致子列表行为未定义
- **`List.of()`**（JDK9）：真正不可变，不能 add/remove

## 3. Set

| 实现 | 特点 |
|------|------|
| `HashSet` | 基于 `HashMap`，元素存 key，value 为固定 `PRESENT` |
| `LinkedHashSet` | 哈希 + 双向链表维护插入顺序 |
| `TreeSet` | 红黑树，有序，需 `Comparable` 或 `Comparator` |

- **equals / hashCode 契约**：相等对象 hash 必同；重写 equals 必须重写 hashCode
- **TreeSet** 元素需可比较，且比较结果与 equals 一致（否则“重复”语义混乱）

## 4. Map（最高频）

| 实现 | 有序性 | null key | 线程安全 |
|------|--------|----------|----------|
| `HashMap` | 无 | 允许（JDK8+） | 否 |
| `LinkedHashMap` | 插入/访问顺序 | 同 HashMap | 否 |
| `TreeMap` | 键排序 | 不允许 | 否 |
| `Hashtable` | 无 | 不允许 | 是（过时） |
| `ConcurrentHashMap` | 无 | 不允许 | 是 |

### HashMap（JDK8+）常问

- **结构**：数组 + 链表 / 红黑树（桶内节点 ≥8 且数组长度 ≥64 树化）
- **put 流程**：hash → 桶下标 → 比较 key → 覆盖或挂链表/树 → 超 **负载因子 0.75** 扩容 2 倍
- **hash**：`(h = key.hashCode()) ^ (h >>> 16)` 降低碰撞
- **为什么容量 2 的幂**：`(n-1) & hash` 代替取模
- **线程安全**：`ConcurrentHashMap`（CAS + synchronized 桶头），不要用 `Collections.synchronizedMap` 扛高并发

### 常用 API

- `putIfAbsent`、`computeIfAbsent`（缓存）、`merge`、`getOrDefault`

## 5. Queue / Deque

| 类型 | 实现 | 说明 |
|------|------|------|
| 普通队列 | `ArrayDeque`、`LinkedList` | `ArrayDeque` 比 `Stack` 类更合适做栈 |
| 优先级 | `PriorityQueue` | 小顶堆，**非线程安全**，迭代无序 |
| 阻塞 | `ArrayBlockingQueue`、`LinkedBlockingQueue` | 生产者-消费者；`put`/`take` 阻塞 |
| 无界阻塞 | `LinkedBlockingQueue` 默认 `Integer.MAX_VALUE` | 面试：可能 OOM |
| 同步 | `SynchronousQueue` | 不存元素，直接交接 |

- **BlockingQueue**：`offer`/`poll` 超时；`put`/`take` 阻塞
- **Deque**：`addFirst`/`addLast`、`pollFirst`（JDK21+ 见 SequencedCollection）
- **LinkedBlockingDeque** 实现的是 `BlockingDeque`，是双端阻塞队列；常见 `BlockingQueue` 对比通常是 `ArrayBlockingQueue` vs `LinkedBlockingQueue`

### Deque 普通版 API

| 操作 | 头部抛异常 | 头部返回特殊值 | 尾部抛异常 | 尾部返回特殊值 |
|------|------------|----------------|------------|----------------|
| 插入 | `addFirst(e)` | `offerFirst(e)` | `addLast(e)` | `offerLast(e)` |
| 获取并删除 | `removeFirst()` | `pollFirst()` | `removeLast()` | `pollLast()` |
| 查看不删除 | `getFirst()` | `peekFirst()` | `getLast()` | `peekLast()` |

- `removeFirst/removeLast/getFirst/getLast`：空时抛 `NoSuchElementException`
- `pollFirst/pollLast/peekFirst/peekLast`：空时返回 `null`
- `ArrayDeque` 不允许 `null`；`LinkedList` 允许 `null`，但用作队列时不建议放 `null`

#### Deque 是两个维度的，要么 throw exception, 要么 null
#### BlockingQueue 是三个维度的，要么 throw exception, 要么 null, 加一个 offer, take 永远阻塞

### Queue / BlockingQueue API

| 操作 | 满/空时抛异常 | 满/空时返回特殊值 | 一直阻塞 | 限时等待 |
|------|----------------|--------------------|----------|----------|
| 插入 | `add(e)` | `offer(e)` | `put(e)` | `offer(e, timeout, unit)` |
| 获取并删除 | `remove()` | `poll()` | `take()` | `poll(timeout, unit)` |
| 查看不删除 | `element()` | `peek()` | 无 | 无 |

- `add/remove/element`：满或空时抛异常
- `offer/poll/peek`：满或空时返回 `false`/`null`
- `put/take`：一直阻塞，直到有空间或有元素；等待期间被 interrupt 会抛 `InterruptedException`
- `offer(timeout)/poll(timeout)`：限时等待，超时返回 `false`/`null`；等待期间被 interrupt 会抛 `InterruptedException`

### ArrayDeque vs LinkedList

| 对比 | `ArrayDeque` | `LinkedList` |
|------|--------------|--------------|
| 底层 | 循环数组 | 双向链表 |
| 内存 | 连续数组，额外对象少 | 每个节点都有前后指针，额外对象多 |
| 局部性 | 好，CPU cache 友好 | 差，节点分散 |
| 头尾操作 | 均摊 O(1) | O(1) |
| 随机访问 | 不支持按索引访问 | 支持 `get(index)`，但 O(n) |
| null | 不允许 | 允许 |
| 典型用途 | 栈、队列、双端队列默认首选 | 需要 `List` 语义，或确实要链表节点特性 |

- 面试结论：**做栈/队列/Deque 时优先选 `ArrayDeque`**。它更轻、更快、cache 友好。
- `LinkedList` 虽然头尾插删也是 O(1)，但节点对象和指针开销大，实际性能经常不如 `ArrayDeque`。
- 只有当你同时需要 `List` 接口能力，或者必须允许 `null`，才更可能考虑 `LinkedList`。

### ArrayBlockingQueue vs LinkedBlockingQueue

| 对比 | `ArrayBlockingQueue` | `LinkedBlockingQueue` |
|------|----------------------|-----------------------|
| 底层 | 固定数组 | 链表节点 |
| 容量 | 必须构造时指定，固定有界 | 可指定容量；不指定时接近无界 `Integer.MAX_VALUE` |
| 锁 | 一个锁控制 put/take | putLock 和 takeLock 分离 |
| 内存 | 预分配数组，内存稳定 | 每个元素一个节点，额外对象和 GC 压力更大 |
| 吞吐 | 简单稳定，锁竞争较集中 | 生产和消费可并行，吞吐可能更好 |
| 背压 | 天然有界，适合限制生产速度 | 指定容量才有明确背压 |
| 公平性 | 可选公平锁构造器 | 通常不强调公平性 |

- 面试结论：**想要固定容量、内存稳定、明确背压 → `ArrayBlockingQueue`**。
- **想要生产和消费并发度更高，且能接受链表节点开销 → `LinkedBlockingQueue`**。
- 使用 `LinkedBlockingQueue` 时最好显式指定容量，否则默认容量太大，生产速度长期大于消费速度时可能造成内存压力甚至 OOM。

### LinkedBlockingQueue vs LinkedBlockingDeque

| 对比 | `LinkedBlockingQueue` | `LinkedBlockingDeque` |
|------|-----------------------|-----------------------|
| 接口 | `BlockingQueue` | `BlockingDeque`，也继承 `BlockingQueue` |
| 方向 | 单端 FIFO：尾部入，头部出 | 双端：头尾都可入、可出、可阻塞 |
| 常用 API | `put`、`take`、`offer`、`poll` | `putFirst/putLast`、`takeFirst/takeLast`、`offerFirst/offerLast` |
| 典型语义 | 普通生产者-消费者队列 | 双端任务队列、工作窃取/任务优先级调整的基础结构 |
| 容量 | 可指定；默认接近无界 | 可指定；默认接近无界 |
| 开销 | 链表节点开销 | 链表节点开销 + 双端操作 API 更复杂 |

- 面试结论：**只需要 FIFO 生产者-消费者 → `LinkedBlockingQueue` 更简单**。
- **需要两端都能放/取，或需要阻塞版 Deque 语义 → `LinkedBlockingDeque`**。
- `LinkedBlockingDeque` 可以当 `BlockingQueue` 用，但如果只用普通 FIFO，选择它通常没有必要，API 面更大，语义也更容易混。
- 两者都建议显式指定容量；默认接近无界，任务堆积时都可能带来内存压力。

## 6. Comparable 与 Comparator

- `Comparable`：自然顺序，`compareTo`，在类内部实现
- `Comparator`：外部策略，`Comparator.comparing`、`thenComparing`、`reversed`

## 7. 不可变与包装

- `Collections.unmodifiableList`：视图，原列表改则“不可变”视图也变
- `List.of` / `Map.of`：真不可变
- `Arrays.asList`：固定大小，不能 add/remove，可 set 元素

## 常见考题

**1. ArrayList 和 LinkedList 怎么选？

**答：**
- **ArrayList**：底层数组，**随机访问 O(1)**，尾插均摊 O(1)，中间插入/删除 O(n)。**默认首选**。
- **LinkedList**：双向链表，头尾插删 O(1)，按索引访问 O(n)。仅在**头尾频繁操作**且元素很多时考虑；很多场景 **ArrayDeque** 做队列/栈更合适。
- 面试结论：查多、遍历多 → ArrayList；别因「链表插入 O(1)」盲目选 LinkedList（还要先遍历到位置）。

**2. HashMap 线程不安全表现？

**答：**
- **JDK7**：多线程扩容可能**环形链表死循环**（CPU 100%），已在后续修复。
- **JDK8+**：并发 put 可能**数据丢失、覆盖**；size 不准确；迭代弱一致性。
- **解决**：`ConcurrentHashMap`；或 `Collections.synchronizedMap`（性能差，仅低并发）。

**3. ConcurrentHashMap 1.7 和 1.8 区别？

**答：**
| | JDK7 | JDK8+ |
|---|------|-------|
| 锁粒度 | **Segment 分段锁**（16 段） | **Node 数组 + 每桶 synchronized/CAS** |
| 锁对象 | Segment | 链表/树根节点 |
| 并发度 | 与 segment 数相关 | 更细，冲突时才锁桶 |
| size | 分段统计 | baseCount + CounterCell |

**4. HashSet 如何保证不重复？

**答：**
- 底层 **HashMap**，元素作 **key**，value 为固定 `PRESENT` 对象。
- 添加时：`hash(key)` → 找桶 → **`equals` 比较**已有 key → 相同则不再插入。
- 故必须保证：**equals 相等则 hashCode 必相等**；重写 `equals` 必须重写 `hashCode`。

**5. PriorityQueue 能放 null 吗？

**答：** **不能**。插入 `null` 抛 `NullPointerException`（堆算法需要比较元素）。需要 null 语义用 `Optional` 包装或自定义比较逻辑。

**6. fail-fast 和 fail-safe？

**答：**
- **fail-fast**（如 ArrayList 迭代器）：迭代中**结构性修改**集合（非迭代器自己的 `remove`）→ 尽快抛 `ConcurrentModificationException`（用 `modCount` 检测）。
- **fail-safe**（如 `CopyOnWriteArrayList`、Concurrent 集合迭代器）：基于**快照或弱一致性**，不抛 CME，可能读不到最新写入。
- **增强 for** 底层是迭代器，并发修改 ArrayList 会 fail-fast。

**7. ArrayDeque 和 LinkedList 怎么选？

**答：**
- 如果只是做栈、队列、双端队列，**默认选 `ArrayDeque`**：数组结构更紧凑，局部性好，头尾操作均摊 O(1)。
- `LinkedList` 每个元素都是节点对象，带前后指针，内存和 GC 开销更大；按索引访问也是 O(n)。
- 只有需要 `List` 接口、允许 `null`、或明确依赖链表特性时，才考虑 `LinkedList`。

**8. ArrayBlockingQueue 和 LinkedBlockingQueue 怎么选？

**答：**
- **`ArrayBlockingQueue`**：固定数组、有界、内存稳定，适合需要明确背压的生产者-消费者模型。
- **`LinkedBlockingQueue`**：链表结构，put/take 使用不同锁，生产和消费可更好并行；但节点对象多，GC 压力更大。
- `LinkedBlockingQueue` 不传容量时默认接近无界，面试要强调：**生产快于消费时可能 OOM，实际使用建议指定容量**。

## 虚拟线程与 Pinning(钉住)核心知识点

### 1. 基本背景

虚拟线程(Virtual Thread,JDK 21 通过 JEP 444 正式发布)是 JVM 管理的轻量级线程,大量虚拟线程会被调度到少量"载体线程"(carrier thread,本质是平台线程/OS 线程)上执行。这个调度器底层是一个 `ForkJoinPool`,默认并行度等于 CPU 核数,可通过 `jdk.virtualThreadScheduler.parallelism` 等系统属性调整。

虚拟线程的核心机制是"挂载(mount)/卸载(unmount)":当虚拟线程执行阻塞操作(如网络 I/O、`ReentrantLock.lock()`、`Thread.sleep()`)时,JVM 会把它从载体线程上卸下来,让载体线程去执行别的虚拟线程;阻塞结束后再重新挂载到某个载体线程继续执行。这就是虚拟线程能用极少量 OS 线程支撑海量并发任务的原理。

### 2. 什么是 Pinning

Pinning 指的是:虚拟线程在执行阻塞操作时**无法被卸载**,只能一直占用着当前的载体线程,直到阻塞结束。这时载体线程会像传统平台线程一样被真正阻塞住,造成资源浪费——这恰恰是虚拟线程想要避免的情况。

注意一个细节:被 pin 住本身不一定是问题,只有"被 pin 住 + 同时发生阻塞"才会真正占用载体线程。

### 3. 导致 Pinning 的常见原因(高频考点)

- **`synchronized` 关键字**:虚拟线程在 `synchronized` 代码块或方法内执行阻塞操作时会被 pin。这是因为 JVM 的监视器(monitor)实现与 OS 线程绑定,无法在持锁状态下安全地切换载体线程。
  - 这是 JEP 444 发布时最大的"坑",当时官方建议把 `synchronized` 替换为 `java.util.concurrent.locks.ReentrantLock`。
  - JDK 24 通过相关改进基本解决了这个问题,`synchronized` 不再普遍导致 pinning(这是版本演进类考点,经常拿来问"现在还需要避免 synchronized 吗")。

- ReentrantLock 从设计上就不会导致 pinning,这正是它被推荐替代 synchronized 的原因。JDK 24 之后 synchronized 也不再 pin 了。目前唯一还会 pin 的场景是虚拟线程在 native 代码帧里发生阻塞,这跟用哪种锁无关,是结构性限制,JDK 26 也没解决。

- **native 方法 / JNI 调用 / Foreign Function & Memory API**:当虚拟线程进入 native 帧后再发生阻塞,无法卸载,因为 native 栈帧仍然依附在 OS 线程栈上。这个问题至今仍存在,是结构性限制。

### 4. 如何检测 Pinning(实操类考点)

- JVM 参数:`-Djdk.tracePinnedThreads=full`(打印完整堆栈)或 `short`(只打印关键帧),在虚拟线程因 pinning 阻塞时输出诊断信息。
- JFR(Java Flight Recorder)事件:`jdk.VirtualThreadPinned`,可在生产环境中低开销采集。

### 5. Pinning 带来的影响

- 大量虚拟线程同时被 pin 并阻塞 → 占满载体线程池(ForkJoinPool 容量有限)→ 其他虚拟线程排队甚至"饿死",并发能力退化到接近平台线程的水平。
- 这也解释了为什么虚拟线程**不适合 CPU 密集型任务**——本来设计目标就是 I/O 密集场景。

### 6. 解决/规避方案

- 用 `ReentrantLock` 替代 `synchronized`(在旧版本上是关键应对手段)。
- 避免在持锁状态下做阻塞 I/O。
- 对必须用 native 调用的场景,尽量减少阻塞时长,或评估是否值得用虚拟线程。
- 关注 JDK 版本演进,JDK 24 之后 `synchronized` 引起的 pinning 大幅减少。

### 7. 容易关联到一起问的相关知识点

- 虚拟线程不应该被池化(`Executors.newVirtualThreadPerTaskExecutor()`,每任务一个虚拟线程,即用即抛)。
- `ThreadLocal` 在虚拟线程场景下的隐患(数量巨大可能导致内存问题),以及 `ScopedValue` 作为替代方案。
- 结构化并发(Structured Concurrency)与虚拟线程的关系。
- 虚拟线程栈存储在堆上、可动态伸缩,与平台线程固定大小栈的区别。
- `Thread.ofVirtual()` / `Thread.ofPlatform()` 新的 Thread Builder API,以及 `Thread.currentThread().isVirtual()` 判断方式。

如果是面试场景,面试官最常追问的链路通常是:**"虚拟线程为什么快" → "什么情况下会失效/退化" → "pinning 是什么、怎么排查、怎么解决"**,这是一个完整的考察链,建议把这条线串起来理解,而不是孤立记忆每个点。

### 8, 当发生 IO blocking 的时候， JVM 会:
- Suspends the virtual thread
- Releases the carrier thread
- Lets another virtual thread run

### Why Is It Faster?
- Much Cheaper Thread Creation
- Less Context Switching
- Better CPU Utilization During I/O

### 面试要记得的陈述
- Virtual threads are faster because blocking no longer consumes an OS thread. The JVM can park a virtual thread during I/O and reuse the underlying carrier thread for other work, drastically improving scalability, reducing memory usage, and minimizing thread-management overhead. They improve concurrency for I/O-bound applications rather than making CPU execution itself faster.

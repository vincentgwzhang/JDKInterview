# JDK 22 面试要点（2024）

## 1. 未命名变量 Unnamed Variables `_`（预览，JDK23+ 迭代）

- 忽略不用的变量：`catch (Exception _) { }`、`for (var _ : list)`
- 提高可读性，**不是**通配符类型

## 2. String Templates（预览，后续撤回）

- `STR."Hello \{name}"` 是 JDK21/22 的预览语法
- JDK23 起没有继续交付该语法；后续 JDK 不能再运行这种写法
- 面试只需了解：这是一个被预览、讨论、然后撤回重做的特性，不要当作现代 Java 可用语法来背

## 常见考题

**1. `_` 与 lambda 中未使用参数的区别？

**答：**
- **`_`（未命名变量）**：语言级占位符，表示**故意忽略**该值；可用于 `catch`、`for`、解构等；同一作用域可多个 `_`。
- **Lambda 未使用参数**：需写 `(a, b) -> ...` 或 IDE 警告；有的用 `_` 作参数名（JDK22+ 正式未命名变量，与旧版「用 _ 作普通标识符」不同）。
- `_` **不是**泛型通配符；不能读取 `_` 绑定的值。

**2. String Templates 相比字符串拼接的优势？

**答：**
- 先说明状态：`STR."Hello \{name}"` 只在 JDK21/22 预览过，后续版本已撤回，不能当作稳定特性使用。
- **类型安全**：插值表达式在编译期检查，非任意拼接。
- **可读性**：`STR."Hello \{name}"` 比 `"Hello " + name` 清晰。
- **防注入**：模板处理器可区分 SQL/HTML 等上下文（预览特性，API 随版本演进）。
- 面试结论：了解设计目标即可；实际项目仍用 `+`、`StringBuilder`、`formatted`、模板引擎或框架提供的安全 API。

**3. FFM API 解决 JNI 哪些痛点？

**答：**
- **Foreign Function & Memory API**（`java.lang.foreign`）：用 Java 安全访问堆外内存、调用 C 函数。
- 相对 JNI：减少手写 `.h`、降低 JVM 崩溃风险、更清晰的内存生命周期（`Arena`）、与 Panama 项目统一。
- 适合高性能库绑定；JNI 仍在维护，FFM 是现代化替代方向。

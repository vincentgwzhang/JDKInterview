# JDK 17 面试要点（2021，LTS）

## 1. Sealed Classes 转正

- 子类必须 **final / sealed / non-sealed** 之一

## 2. Switch 模式匹配（预览，JDK21 转正）

- `case String s`、`case Integer i` 等
- 与 sealed 类型联合做穷尽检查

## 3. 恢复始终严格的浮点语义（了解）

- 移除 `--illegal-access` 等过渡开关

## 4. 新随机数 API

- `RandomGenerator` 及多种实现 `RandomGeneratorFactory`

## 5. 其他

- **Foreign Function & Memory API**（孵化器，后续演进）
- **Deprecate Security Manager**

## 常见考题

**1. 为什么 17 是企业升级 11 后的常见目标？

**答：**
- **LTS**（支持到 2029+），补丁与生态成熟。
- **语言定稿**：Sealed Classes、Records、instanceof 模式匹配稳定可用。
- **性能**：GC、JIT、容器感知持续改进。
- **框架对齐**：Spring Boot 3 基线 Java 17；大量库已放弃 11 以下。
- 11→17 跳跃小于直接上 21，许多企业作为**中间 LTS** 站。

**2. Sealed 子类三种修饰符含义？

**答：**
| 修饰符 | 含义 |
|--------|------|
| `final` | 不能再被继承 |
| `sealed` | 继续指定 `permits`，子类仍受控 |
| `non-sealed` | 解除密封，允许任意类继承该子类 |

**3. `RandomGenerator` 相比 `Random` 的优势？

**答：**
- **统一接口** + `RandomGeneratorFactory` 按名称获取多种算法（L32X64MixRandom、Xoshiro256PlusPlus 等）。
- 算法可插拔，性能与统计特性可选；线程安全实现与本地随机分离更清晰。
- `Random` 仍可用；新代码推荐 `RandomGenerator.of("...")` 或 `ThreadLocalRandom.current()` 的现代化替代。

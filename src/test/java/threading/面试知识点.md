# Java 多线程 / 并发 面试要点

> 聚焦**高频考点**，与 `jdk21` 虚拟线程章节互补；此处以经典 `java.util.concurrent` 为主。

## 1. 线程基础

- **创建方式**：`Thread`、`Runnable`、`Callable` + `FutureTask`、线程池（推荐）
- **六种状态**：NEW → RUNNABLE → BLOCKED / WAITING / TIMED_WAITING → TERMINATED
- **start vs run**：`start()` 启动新线程；`run()` 只是普通方法调用
- **interrupt**：协作式取消；`interrupt()` 设标志，阻塞方法抛 `InterruptedException`，应 `interrupt()` 恢复标志或退出

## 2. synchronized 与锁

- **作用**：互斥 + 可见性（释放锁 happens-before 后续获取同一把锁）
- **锁对象**：实例方法锁 `this`，静态方法锁 `Class`，代码块锁指定对象
- **可重入**：同一线程可多次获取同一把监视器锁
- **与 Lock**：`ReentrantLock` 可中断锁、超时、`tryLock`、公平锁；需 `finally unlock`

## 3. volatile

- **保证**：可见性、禁止指令重排序（对 volatile 写 happens-before 后续 volatile 读）
- **不保证**：复合操作原子性（如 `i++`）
- **场景**：状态标志、双重检查锁中的实例引用（需配合 synchronized）

## 4. wait / notify / notifyAll

- 必须在 **已持有监视器** 的 synchronized 块内调用
- `wait()` 释放锁并进入 WAITING；`notify()` 唤醒一个等待线程
- **经典题**：为什么用 `while` 而不是 `if` 检查条件？—— 虚假唤醒、多消费者竞争

## 5. JUC 工具类

| 类 | 用途 |
|----|------|
| `CountDownLatch` | 一个或多个线程等待其他线程完成（减计数） |
| `CyclicBarrier` | 一组线程互相等待到齐再继续（可循环） |
| `Semaphore` | 控制并发数量（许可） |
| `ReentrantLock` + `Condition` | 多条件队列（如两个 Condition 实现阻塞队列） |

## 6. 线程池

- **核心参数**：`corePoolSize`、`maximumPoolSize`、`workQueue`、`RejectedExecutionHandler`
- **流程**：任务 → 核心线程 → 队列 → 非核心线程 → 拒绝策略
- **常见池**：`FixedThreadPool`、`CachedThreadPool`、`ScheduledThreadPool`；生产用 `ThreadPoolExecutor` 显式配置
- **面试**：为什么不推荐 `Executors.newFixedThreadPool`？—— 队列无界可能 OOM

## 7. ThreadLocal

- 每个线程一份副本；底层 `ThreadLocalMap`（弱引用 key）
- **内存泄漏**：key 被回收后，value 仍被线程持有直到线程销毁；用完 `remove()`
- **虚拟线程**：大量虚拟线程 + ThreadLocal 可能占内存，优先考虑 `ScopedValue`（JDK 21+）

## 8. 原子类与 CAS

- `AtomicInteger` 等：`compareAndSet` 乐观锁，失败自旋
- **ABA 问题**：`AtomicStampedReference`（了解）
- **与 synchronized**：低竞争 CAS 性能好；高竞争自旋浪费 CPU

## 9. ConcurrentHashMap（高频）

- JDK8+：**CAS + synchronized 桶头节点**，分段锁已移除
- 常用：`putIfAbsent`、`computeIfAbsent`（缓存场景）
- **不允许** null key/value

## 10. CompletableFuture

- 异步编排：`supplyAsync`、`thenApply`、`thenCompose`、`allOf`、`anyOf`
- 默认线程池 `ForkJoinPool.commonPool()`，可指定 `Executor`
- 异常：`exceptionally`、`handle`

## 11. 死锁

- **四个必要条件**：互斥、占有且等待、不可抢占、循环等待
- **避免**：固定加锁顺序、超时 `tryLock`、银行家算法（了解）
- **排查**：`jstack`、线程 dump 看 `Found one Java-level deadlock`

## 12. 虚拟线程（简表，详见 jdk21）

- **适合** I/O 密集；**不适合** 长时间 CPU 密集或大量 synchronized（pin 载体线程，JDK 持续改进）
- **不要**池化虚拟线程；用 `newVirtualThreadPerTaskExecutor`

## 常见考题

**1. synchronized 和 Lock 区别？

**答：**
| | synchronized | ReentrantLock |
|---|--------------|---------------|
| 层面 | JVM 关键字，自动释放 | API，需 `unlock()` |
| 可中断 | 否 | `lockInterruptibly` |
| 超时 | 否 | `tryLock(timeout)` |
| 公平锁 | 非公平 | 可选公平 |
| 条件队列 | 单一 wait-set | 多个 `Condition` |
| 异常 | 自动释放锁 | 必须在 `finally` 释放 |

**2. sleep 和 wait？

**答：**
| | `Thread.sleep` | `Object.wait` |
|---|----------------|---------------|
| 所属 | Thread 静态方法 | Object 实例方法 |
| 释放锁 | **不释放** | **释放**当前对象的监视器锁 |
| 使用 | 任意位置 | 必须在 **synchronized** 块内 |
| 唤醒 | 超时或 interrupt | `notify`/`notifyAll` 或超时 |
| 用途 | 暂停执行 | 线程协作、等待条件 |

**3. 如何保证线程安全？

**答：**
1. **不可变对象**（final 字段、Record、无发布可变状态）。
2. **互斥**：synchronized、Lock，临界区串行化。
3. **线程封闭**：数据不跨线程共享（ThreadLocal、栈内变量）。
4. **并发容器**：`ConcurrentHashMap`、`CopyOnWriteArrayList` 等。
5. **原子类 / CAS**：`AtomicInteger`，适合简单计数、状态位。
6. **并发工具**：CountDownLatch、Semaphore 等协调，而非共享可变数据。

**4. 线程池参数如何设？

**答：**
- **CPU 密集**：线程数 ≈ **CPU 核数 + 1**（减少上下文切换）。
- **I/O 密集**：可 **远大于核数**（线程阻塞时不占 CPU），或用 **虚拟线程**（JDK21+）。
- **队列**：有界队列防 OOM；`LinkedBlockingQueue` 默认容量极大需谨慎。
- **拒绝策略**：生产常用 `CallerRunsPolicy`（背压）或自定义记录+告警；避免无声丢弃。
- **监控**：队列长度、活跃线程、拒绝次数。

**5. happens-before 规则？

**答：** JMM 保证可见性的偏序关系，无需额外同步即可安全读：
1. **程序次序**：同线程内，前面的操作 hb 后面的操作。
2. **监视器锁**：unlock hb 后续对同一锁的 lock。
3. **volatile**：对同一变量的写 hb 后续对该变量的读。
4. **线程启动**：`start()` hb 线程内动作。
5. **线程 join**：线程内动作 hb `join()` 返回。
6. **传递性**：A hb B，B hb C → A hb C。
